# Batch 1 Candidates Spec

## Goal

Batch 1 exists to produce the first approved set of:

- taxonomy candidates
- use-case slices
- hero case

This batch does not approve final production truth. It approves the first controlled demo set.

## 1. Partner/channel taxonomy

### Taxonomy shape

Three layers:

- data-source partners
- acquisition/distribution channels
- shared-data partnership models

### Data-source candidates

- Must stay anchored to the 4 SF8 source families:
  - telco
  - social
  - e-wallet
  - e-commerce
- Can include equivalent public benchmarks when core public references are not enough

### Acquisition/distribution channels

- Must be easy to explain in a demo
- Must remain close to consumer finance / embedded finance logic

### Shared-data partnership models

- Must distinguish:
  - data-sharing
  - distribution-only

## 2. Signal taxonomy

### Signal groups

- external behavioral signals
- external profile/context signals
- partner/channel-derived signals
- early reaction signals

### Signal granularity

Each signal candidate must include:

- feature-level grouping
- example fields
- scoring intent

### Early reaction scope

Must support:

- open
- click
- ignore
- response speed
- offer category preference
- channel preference

Signals may be grouped into scoring-friendly buckets if still traceable.

## 3. Product groups and packages

### Product groups

- Total groups: `4`
- Candidate method:
  - public/reference aligned
  - SF8-fit first
  - still mappable to market-like package naming

### Group naming

Use `dual naming`:

- internal business-generic group name
- near-market display label

### Product packages

Use `dual naming`:

- internal canonical name
- public/market-like display label

### Package depth

Each group must have:

- minimum 2 package candidates
- maximum 4 package candidates

## 4. Archetypes

### Count

- `6` candidates in Batch 1

### Design basis

Each archetype must be based on:

- behavioral patterns
- partner/channel context
- early reaction posture

### Required fields

- name
- behavioral profile
- partner/channel context
- early reaction posture
- expected product tendency
- expected action tendency

## 5. Candidate evidence requirement

Every candidate must include:

- source link
- extraction note
- reason for inclusion
- confidence/fit note

## 6. Candidate status model

- draft
- pending review
- approved
- published
- rejected

## 7. Review format

Review output must be:

- table
- short note
- `recommended approve / hold / reject`

## 8. Use-case slices

Batch 1 must produce:

- `1 strong-fit slice`
- `1 borderline / interesting slice`

### Each slice must include

- 1 partner/channel setup
- 1 signal bundle
- 1 archetype
- 1 product outcome
- 1 action outcome
- 1 what-if path

## 9. Hero case

Hero case selection rule:

- choose the case with the best score

## 10. Batch 1 completion condition

Batch 1 is complete only when:

- taxonomy candidates are approved
- 2 use-case slices are approved
- hero case is selected
